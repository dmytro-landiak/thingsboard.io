# TBMQ Documentation Improvement — Design Spec

**Owner:** dlandiak

## Goal

Systematically improve every TBMQ documentation page (CE + PE) for grammar/clarity, technical correctness against the TBMQ source repos, MQTT-spec accuracy, command executability, and diagram coverage. The pass is content-quality only — no new features, no refactors, no new components, no new deps.

## Scope

In scope: **shared include files** under `src/content/_includes/docs/mqtt-broker/**` plus their corresponding CE/PE stub entry points under `src/content/docs/docs/mqtt-broker/**`. ~72 distinct guides across 13 sections.

Out of scope (explicitly excluded by user on 2026-05-25):

- **Installation guides** (the entire "Installation" sidebar group — Docker, Docker Compose, Kubernetes/Helm clusters across AWS/Azure/GCP/Minikube, live demo, upgrade instructions, how-to-change-config, building-from-source — 16 files).
- **Configuration reference stubs** — `installation/config.mdx` and `installation/ie-config.mdx` for CE and PE. These are **auto-generated by a script**; do not hand-edit.

## Hard rules (apply to every guide)

These are derived from `CLAUDE.md` and project memory and supersede general writing preferences:

1. **No "TBMQ CE" / "TBMQ PE" in body prose.** The edition is clear from the navigation menu — use **"TBMQ"** alone.
2. **Sentence case for table column values**, never title case.
3. **Preserve every command, code block, config snippet, env var, CLI flag, YAML key, and curl invocation verbatim.** Only improve the descriptive prose around them. If a command is technically wrong, fix it from source — do not "polish" commands that are already correct.
4. **Never claim a feature is "UI only".** Every TBMQ UI action has a matching REST endpoint. If a guide says otherwise, fix it.
5. **Use existing components** (`DocImage`, `Banner`, `DocLink`, `YouTubeVideo`, `ImageGallery`, code-block meta options) rather than raw HTML/markdown where appropriate. Reference the `edit-doc` skill for component rules.
6. **Never hardcode version strings.** Import from `~/data/versions` (`TBMQ_VER`, `TBMQ_PE_VER`).
7. **Indentation:** tabs in `.mdx`; spaces in YAML/JSON/MD frontmatter.
8. **No new files** except this spec, the companion plan, the per-section reports tracked in the plan, and `tbmq-docs-future-work.md` at Phase 3.
9. **No commits or pushes.** The user reviews `git diff` and commits manually.
10. **TBMQ source repos at `~/projects/tbmq` and `~/projects/tbmq-pe` are read-only references.** Never edit, commit, or run mutating git commands there.

## Per-guide sub-agent brief (template)

This is the verbatim brief each sub-agent receives (the `{include path}`, `{ce stub}`, `{pe stub}` placeholders get substituted at dispatch time):

> You are improving one TBMQ doc page: **{absolute include path}**. The CE stub at {ce stub} and the PE stub at {pe stub} both import this include — your edit applies to both products. Do not edit the stubs themselves unless their frontmatter is wrong.
>
> **Hard project rules** — see "Hard rules" section in `docs/superpowers/specs/2026-05-25-tbmq-docs-improvement-design.md`.
>
> **Tasks (in this order):**
>
> 1. **Read the guide top to bottom.** Note structure, intent, target audience, and any conditional product blocks (`{product === Products.TBMQ_PE && ...}`).
>
> 2. **Grammar & clarity pass.** Fix grammar, awkward phrasing, ambiguous pronouns, redundant words, run-on sentences. Improve flow. Bar for any edit: "would a non-native English reader understand this faster after the change?" If no, leave it alone. Do not rewrite for the sake of rewriting.
>
> 3. **Technical-correctness pass against TBMQ source.** Use `Read`, `Grep`, `Glob` on the local checkouts. **Do not WebFetch GitHub.**
>    - TBMQ CE: `~/projects/tbmq`
>    - TBMQ PE: `~/projects/tbmq-pe`
>
>    Run `git -C ~/projects/tbmq status` and `git -C ~/projects/tbmq-pe status` once at the start; flag the branch and staleness in your report. Do not `git pull` or mutate either checkout.
>
>    For every claim in the guide:
>    - **Config parameters** — locate the actual key in `application.yml`, `application-*.yml`, or `helm/values.yaml`; confirm name, default, and behavior. PE-only parameters live only in `~/projects/tbmq-pe`.
>    - **REST endpoints** — `Grep` for the path in `*Controller.java` in both repos; confirm method, path variables, request body, response body, and required permissions.
>    - **UI steps** — read the relevant Angular component/template under `ui-ngx/` (or equivalent); confirm menu labels and field names match the doc.
>    - **CE vs PE feature gating** — verify by checking which repo holds the relevant code. Many features exist in both with extended PE behavior; reflect that nuance.
>
> 4. **MQTT spec pass.** If the guide references MQTT semantics (QoS, sessions, retained, LWT, MQTT 5 properties, topic alias, shared subs, flow control, reason codes), verify against:
>    - MQTT v3.1.1: `https://docs.oasis-open.org/mqtt/mqtt/v3.1.1/os/mqtt-v3.1.1-os.html`
>    - MQTT v5.0: `https://docs.oasis-open.org/mqtt/mqtt/v5.0/os/mqtt-v5.0-os.html`
>
>    Cite the spec section in your final report, not in the doc body.
>
> 5. **Command-executability pass.**
>    - CLI: correct flags, paths, ordering, shell quoting for the target OS.
>    - `docker`/`docker-compose`: tag matches `~/data/versions`; ports, env, volumes correct.
>    - `kubectl`/`helm`: matches current chart version; correct namespace flags.
>    - `curl`: correct path, method, headers (`Content-Type`, `X-Authorization`), JSON body shape.
>
>    Fix wrong commands from source. **Never invent a command you cannot verify** — flag instead.
>
> 6. **Diagram placeholders.** Where a diagram would aid comprehension (architecture, message lifecycle, cluster topology, sequence flows), insert:
>
>    ````mdx
>    {/* DIAGRAM-PLACEHOLDER: short description of what to draw */}
>
>    ```text
>    [ASCII art ~60 cols wide]
>    ```
>    ````
>
>    These will be replaced later by proper rendered diagrams.
>
> 7. **Self-verify** before reporting done (per `superpowers:verification-before-completion`):
>    - Re-read the final diff.
>    - Confirm none of the hard rules were violated.
>    - Confirm every claim you changed traces back to a cited source.
>
> **Do not run `pnpm build` or `pnpm build:fast`.** You may run `pnpm check` if your edits change MDX structure or component usage.
>
> **Final report format:**
>
> ```
> File: {include path}
> Grammar/clarity edits: N
>   - short note per spot
> Technical fixes: N
>   - each with the source file path you checked
> MQTT spec fixes: N
>   - each citing spec §x.y.z
> Command fixes: N
> Diagram placeholders added: N (list section headings)
> Could-not-verify: list anything you flagged
> Open questions for the user: ...
> ```

## Workflow

```
Phase 0 (done): brainstorming → writing-plans → dispatching-parallel-agents → verification-before-completion
Phase 1 (done): discover & group, produce section table, get user approval
Phase 2 (this is what we execute): one section at a time, gated by user "go"
  └─ For each section:
       ├─ Dispatch one sub-agent per guide (cap 5 concurrent, replace as they finish)
       ├─ Each sub-agent: 7-task brief above
       ├─ After all guides done: per-guide summary table appended to the plan
       └─ Ask user to approve next section
Phase 3 (final): Write `tbmq-docs-future-work.md` at repo root with P0/P1/P2 prioritization
  └─ Cover: gaps in current docs, MQTT-related additions, TBMQ-specific additions, cross-product additions
```

## Rules of engagement

- **Never commit or push.** User reviews `git diff` and commits.
- **One section at a time** in Phase 2; always stop and ask before the next.
- **Within a section, parallelize** with sub-agents (cap 5 concurrent).
- **Honesty over completeness.** If a sub-agent can't verify a technical claim (PE-only code with no matching source, ambiguous spec), it flags it — never silently changes it.
- **No refactors, no new components, no new deps.** Content-quality pass only.
