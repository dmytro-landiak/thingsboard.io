An MQTT session is the server-side state TBMQ maintains for a connected client. It holds the client's active subscriptions,
pending QoS 1 and QoS 2 message acknowledgements, and — for persistent sessions — messages queued while the client was offline.
Understanding sessions is a prerequisite for reasoning about message delivery guarantees, reconnect behavior, and resource usage.

Whether that state survives a disconnect is what separates a **persistent session** from a **non-persistent** (clean) one.
Getting this choice right matters: persistent sessions guarantee no missed messages at the cost of broker-side storage,
while non-persistent sessions are stateless and lightweight but lose undelivered messages on disconnect.

## Session state

Every active session holds the following state, regardless of type:

| State | What it contains |
|-------|-----------------|
| **Subscriptions** | All topic filters the client has subscribed to, with their QoS levels |
| **In-flight messages** | QoS 1 and QoS 2 messages sent to the client but not yet acknowledged |
| **Pending messages** | QoS 1 and QoS 2 messages waiting to be sent to the client |
| **QoS 2 receive state** | Packet IDs of QoS 2 publishes received from the client but not yet released (`PUBREL` pending) |

Persistent sessions additionally retain this state across disconnects, so it is restored when the client reconnects.

## Persistent vs non-persistent sessions

### Non-persistent session

A session is non-persistent when the `CONNECT` packet meets any of these conditions:

| MQTT version | Condition |
|---|---|
| MQTT 3.1.1 | `Clean Session = 1` |
| MQTT 5.0 | `Clean Start = 1` and `Session Expiry Interval = 0` (or not set) |

<ImageGallery images={[{ src: '/src/assets/schemas/non-persistent-session.svg', alt: 'Non-persistent session lifecycle: on connect the session is created with subscriptions set and messages delivered; on disconnect the session is destroyed, subscriptions are lost, and queued messages are discarded.', caption: 'A non-persistent session exists only for the duration of the connection: state is created on connect and destroyed on disconnect.' }]} />

TBMQ creates session state for the duration of the connection and discards it immediately on disconnect.
No server-side state is kept between connections, and the next CONNECT creates a fresh session from scratch.

**Use when:** the client always resubscribes on connect, does not need to receive messages published while it was offline,
and you want to minimize broker memory usage. Typical examples: dashboards, short-lived CLI tools, and stateless microservices
that always start clean.

### Persistent session

A session is persistent when the `CONNECT` packet meets any of these conditions:

| MQTT version | Condition |
|---|---|
| MQTT 3.1.1 | `Clean Session = 0` |
| MQTT 5.0 | `Session Expiry Interval > 0` (regardless of `Clean Start`) |
| MQTT 5.0 | `Clean Start = 0` and `Session Expiry Interval = 0` (or not set) |

<ImageGallery images={[{ src: '/src/assets/schemas/persistent-session.svg', alt: 'Persistent session lifecycle: on connect the session is created or resumed with subscriptions active; on disconnect the session is kept and subscriptions are preserved; on reconnect the session is restored and queued messages are delivered.', caption: 'A persistent session survives disconnects: subscriptions are preserved on the broker and any messages queued while the client was offline are delivered on reconnect.' }]} />

TBMQ retains subscriptions and queued messages across the disconnect. On reconnect, the broker resumes the session,
so the client does not need to resubscribe and any messages that arrived while it was offline are delivered in order.

**Use when:** the client must not miss messages during outages or restarts. Typical examples: IoT devices on unreliable
networks, backend services that process every event, and rule engines that must consume a complete stream.

<Aside type="note">
  In MQTT 5.0, `Session Expiry Interval` controls how long the session survives after disconnect: set it to `0` for
  immediate cleanup, to a number of seconds for timed expiry, or to `0xFFFFFFFF` to request that the session never
  expire. TBMQ caps any client-requested value at the broker's `MQTT_CLIENT_SESSION_EXPIRY_MAX_EXPIRY_INTERVAL`
  (default `604800` seconds, i.e. one week).

  In MQTT 3.1.1, a `Clean Session = 0` session stays on the broker until the client reconnects with `Clean Session = 1`
  or until the administrative cleanup TTL elapses (`MQTT_CLIENT_SESSION_EXPIRY_TTL`, default `604800` seconds; set to
  `0` to disable). The same TTL also applies to MQTT 5.0 sessions created with `Clean Start = 0` and
  `Session Expiry Interval = 0`.
</Aside>

## How TBMQ stores persistent session state

The storage backend for a persistent session depends on the <DocLink product={props.product} path="concepts/client-types">client type</DocLink>:

| | **DEVICE client** | **APPLICATION client** |
|---|---|---|
| **Subscription state** | In-memory on the broker node, replicated via Kafka | In-memory on the broker node, replicated via Kafka |
| **Offline message queue** | Redis | Dedicated Kafka topic (`tbmq.msg.app.<client-id>`) |
| **Queue depth limit** | `MQTT_PERSISTENT_SESSION_DEVICE_PERSISTED_MESSAGES_LIMIT` (default `10000`) | Kafka topic retention settings (`retention.ms`, `retention.bytes`) |
| **Delivery on reconnect** | Replayed from Redis by the reconnecting node | Replayed from the client's Kafka topic |

{/* DIAGRAM-PLACEHOLDER: side-by-side flow for DEVICE vs APPLICATION offline message storage and replay on reconnect */}

```text
  DEVICE client (persistent)              APPLICATION client (persistent)
  ----------------------------            -------------------------------
  Publisher  -->  Broker node             Publisher  -->  Broker node
                     |                                       |
                     v                                       v
                 [ Redis ]                          [ Kafka topic:
              (sorted set per                       tbmq.msg.app.<id> ]
               client, capped)                     (durable log, retention
                     |                              by time + size)
                     v                                       |
              Reconnect (any node) <---------- Reconnect (any node)
                     |                                       |
                     v                                       v
              Replay from Redis                  Replay from Kafka offset
```

Redis gives DEVICE clients fast, low-latency queue reads and writes suitable for moderate per-client message volumes.
APPLICATION clients get a durable Kafka log that scales to high throughput and long retention without memory pressure.

## When to use persistent sessions

| Situation | Recommendation |
|-----------|---------------|
| IoT device on a cellular or satellite link | Persistent — connections drop frequently |
| Backend analytics service | Persistent — must consume every event |
| Mobile app or browser dashboard | Non-persistent — resubscribes on every connect; stale messages are not useful |
| Load testing or benchmarking client | Non-persistent — no need for state between runs |
| APPLICATION client (any) | Persistent — a non-persistent APPLICATION session has no Kafka topic and therefore no offline buffer |

<Aside type="caution">
  An APPLICATION client with a non-persistent session receives messages only while connected, because no dedicated
  Kafka topic is created and no offline buffer exists. TBMQ displays a warning on the session details page when this
  condition is detected.
</Aside>

## Session visibility

Both active and disconnected persistent sessions are listed on the **Sessions** page of the TBMQ UI, and the same
data is available through the REST API. From there you can inspect subscriptions and connection metadata, or clear
a session manually if needed. For details, see <DocLink product={props.product} path="user-guide/ui/sessions">Sessions</DocLink>.
