## Running

Deploy TBMQ:

```bash
./k8s-deploy-tbmq.sh
```

Once all resources have started, open `http://{your-cluster-ip}:30001` in your browser (e.g. **http://192.168.49.2:30001**).
Check your cluster IP with:

```bash
minikube ip
```

You should see the TBMQ login page. Use the default **System Administrator** credentials:

**Username:**

```
sysadmin@thingsboard.org
```

**Password:**

```
sysadmin
```

On first login, you are prompted to change the default password and re-login with the new credentials.

## Logs, delete statefulsets and services

To view TBMQ node logs:

1. List running TBMQ pods:

   ```bash
   kubectl get pods -l app=tbmq
   ```

2. Fetch logs for a specific pod:

   ```bash
   kubectl logs -f TBMQ_POD_NAME
   ```

   Where `TBMQ_POD_NAME` is the pod name from the list above.

To check the state of all pods, services, deployments, and statefulsets:

```bash
kubectl get pods
kubectl get services
kubectl get deployments
kubectl get statefulsets
```

See the <a href="https://kubernetes.io/docs/reference/kubectl/cheatsheet/" target="_blank" rel="noopener noreferrer">kubectl Cheat Sheet</a> for more details.

To delete TBMQ nodes:

```bash
./k8s-delete-tbmq.sh
```

To delete all resources including databases:

```bash
./k8s-delete-all.sh
```

### Run upgrade

Pull the latest changes from the release branch:

<Code code={gitPullCmd} lang="bash" />

<Aside type="tip">
  To upgrade to a specific version (e.g. v2.0.0), replace the branch with the target branch name, e.g. <code>release-2.0.0</code>.
</Aside>

<p><strong>Note:</strong> Make sure any custom changes are not lost during the merge.</p>

<Aside>
  <p>If you encounter conflicts during the merge that are unrelated to your changes, accept all remote changes:</p>
  <ol>
    <li>Abort the current merge: <code>git merge --abort</code></li>
    <li>Re-pull accepting remote changes: <code>git pull origin {TBMQ_BRANCH} -X theirs</code></li>
  </ol>
  <p>Merge strategy options:</p>
  <ul>
    <li><code>-X ours</code> — auto-resolve conflicts by favoring your local version</li>
    <li><code>-X theirs</code> — auto-resolve conflicts by favoring the remote version</li>
  </ul>
</Aside>

After pulling, run the upgrade script:

```bash
./k8s-upgrade-tbmq.sh
```

<Aside>
  <p>You may optionally stop TBMQ pods before running the database upgrade to ensure a consistent DB state:</p>
  <Code code="./k8s-delete-tbmq.sh" lang="bash" />
  <p>This causes downtime, but most upgrades do not require stopping TBMQ. Once the upgrade completes, redeploy to roll out the new version:</p>
  <Code code="./k8s-deploy-tbmq.sh" lang="bash" />
</Aside>
