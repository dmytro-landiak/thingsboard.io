**Anomaly detection** identifies unexpected behavior in telemetry data — patterns that differ significantly from what is
considered normal. These anomalies can indicate system malfunctions, equipment degradation, security breaches, or
operational inefficiencies.

Trendz provides an out-of-the-box solution for anomaly detection that automatically finds abnormal patterns in
time-series data without requiring manual thresholding or expert labeling. Advanced users also have full control over
model configuration — input preparation, feature extraction, distance function tuning, and scoring logic — all within
an intuitive interface.

## Key Concepts

Trendz anomaly detection is based on two core metrics:

| Metric | Description |
|--------|-------------|
| **Anomaly Score** | A numeric value representing how far a data segment deviates from expected (normal) behavior. Measures the **intensity** of the anomaly. |
| **Anomaly Score Index** | A composite metric that combines **anomaly score** with the **duration** of the anomaly. Helps prioritize anomalies that have a **greater cumulative impact** over time. |

**Example — pump vibration analysis:**

- **Anomaly A** — a sharp vibration spike lasting ~5 seconds. High anomaly score, but short duration keeps the score
  index low.
- **Anomaly B** — a longer-lasting deviation without large spikes. Lower anomaly score, but the score index is higher
  due to extended duration — suggesting greater long-term impact on pump health.

Use **Anomaly Score** to detect sharp, short-term deviations. Use **Anomaly Score Index** to surface sustained anomalies
that may cause greater damage over time.

## Detection Approach

Trendz uses **unsupervised machine learning** for anomaly detection, based on clustering:

```plaintext
  Raw telemetry
       │
       ▼
  ┌────────────────┐
  │  Segmentation  │  Split data into fixed-length windows
  └────────┬───────┘
           │
           ▼
  ┌────────────────┐
  │  Normalization │  Scale values for comparability
  └────────┬───────┘
           │
           ▼
  ┌────────────────┐
  │ Feature extract│  Describe the shape of each segment
  └────────┬───────┘
           │
           ▼
  ┌────────────────┐
  │   Clustering   │  Group segments into "normal" clusters
  └────────┬───────┘
           │
           ▼
  ┌────────────────────────────────┐
  │  Anomaly Score = distance from │
  │  nearest cluster centroid      │
  └────────────────────────────────┘
```

Unsupervised detection does not require labeled data. Models learn normal behavior by grouping similar data segments
into clusters. Anomalies are segments that deviate significantly from all cluster centroids.

<Aside type="note">
  Trendz currently supports only unsupervised machine learning algorithms for anomaly detection, using clustering-based
  methods such as K-Means, DBSCAN, and Gaussian Mixture Models.
</Aside>

## Tabs Overview

### Anomaly Models Tab

The **Anomaly Models** tab is the entry point for managing all anomaly detection models. Access it by clicking the
*Anomaly Models* icon in the left-side navigation.

<ImageGallery images={[
  {
    src: '/src/assets/images/trendz/anomaly/page-anomaly-models.png',
    alt: 'Anomaly Models tab listing all configured anomaly detection models',
    caption: 'Anomaly Models tab — shows model name, type, status, and configured jobs.',
  },
]} />

Each model row shows:

| Field | Description |
|-------|-------------|
| **Creation Date** | When the model was created |
| **Name** | Model name (editable) |
| **Type** | Algorithm type — currently only *CLUSTERING* is supported |
| **Status** | Current state of the model (see below) |
| **Anomaly autodiscovery** | Whether the refresh job is enabled. See <DocLink product={props.product} path="anomaly/refresh-reprocess">Refresh &amp; Reprocess</DocLink>. |
| **Saving telemetry** | Whether detected anomalies are written back to ThingsBoard. See <DocLink product={props.product} path="anomaly/save-to-tb">Save to ThingsBoard</DocLink>. |
| **Alarm auto creation** | Whether alarms are automatically raised for anomalies. See <DocLink product={props.product} path="anomaly/alarms">Anomaly-Driven Alarms</DocLink>. |

**Model statuses:**

| Status | Meaning |
|--------|---------|
| **READY** | The model is trained and ready for anomaly detection |
| **QUEUED** | Waiting for the training job to start |
| **IN PROGRESS** | Currently being trained |
| **CANCELLED** | Training was canceled — model must be rebuilt |
| **FAILED** | Training failed — model must be rebuilt |

**Available actions:**

- **Create Model** — click *Create model* in the top-right corner to open a new model in the Input tab.
- **Inspect Model** — click any row to open the model in the Input tab.
- **Rename Model** — open the three-dot *Actions* menu and select *Rename*. A text field appears in the Name column; press **Enter** to save.
- **Delete Model** — open the three-dot *Actions* menu and select *Delete*, then confirm.

### Input Tab

The **Input** tab is the primary configuration surface for an anomaly model. It opens automatically when you create
or inspect a model. See <DocLink product={props.product} path="anomaly/build">Model Properties</DocLink> for full configuration details.

<ImageGallery images={[
  {
    src: '/src/assets/images/trendz/anomaly/page-input-tab.png',
    alt: 'Input tab with model configuration options',
    caption: 'Input tab — configure model properties, save changes, or trigger a rebuild.',
  },
]} />

Available actions:

- **Rename** — click the pencil icon next to the model name, enter a new name, and press **Enter**. Click **Save Model** to persist the change. A rebuild is **not** required for renaming alone.
- **Revert** — click **Revert** in the upper-right corner to discard unsaved changes and restore the last saved state.
- **Configure Jobs** — available only for **READY** models. Click **Jobs** to open the jobs configuration popup and set up the refresh job.
- **Change Properties** — update model properties such as segmentation settings, feature extraction, and clustering parameters. Click **Save Model** to apply.

<Aside type="caution">
  Most property changes trigger a model rebuild when you click **Save Model**. Only changes to the **name**,
  **telemetry key**, **storage frequency**, or **alarm configuration** can be saved without rebuilding. Rebuilding
  deletes all existing detected anomalies and disables the refresh job.
</Aside>

### Summary Tab

The **Summary** tab provides an overview of detected anomalies grouped by device.

You can:
- View statistics for **all devices** simultaneously or drill into a **specific device** by selecting it in the table.
- Switch display mode using the **Review**, **Dynamic**, and **Calendar** buttons in the upper-right corner.

**Display modes:**

| Mode | Scope | Description |
|------|-------|-------------|
| **Calendar** | All devices | Total anomaly index per day. Toggle *Full Range* to show the entire year, not just months with anomalies. |
| **Dynamic** | Selected device | Monthly distribution of anomalies. Green bars = total anomaly score index for the device; blue line = average across all devices. |
| **Review** | Selected device | Shows the latest detected anomaly. Zoom out to explore the full anomaly history. |

### Cluster Info Tab

The **Cluster Info** tab shows how the model organized data segments into clusters during training.
Use it to validate that clustering accurately captures normal behavior patterns.

**Cluster table** (bottom of screen):

| Column | Description |
|--------|-------------|
| **Cluster ID** | Unique cluster identifier |
| **Segments Count** | Number of segments closest to this cluster |
| **Segments Percent** | Percentage of total segments in this cluster |
| **Total Duration** | Combined duration of all segments in this cluster |
| **Min Score** | Minimum anomaly score among segments in this cluster |
| **Max Score** | Maximum anomaly score among segments in this cluster |

**Centroid visualization** (top of screen): displays the average pattern (centroid) for each cluster, showing the
typical shape of segments that belong to it.

Click any row in the cluster table to open a detailed view showing the selected centroid alongside 20+ random
member segments — useful for assessing intra-cluster variability and validating clustering accuracy.

### Tasks Tab

The **Tasks** tab lists background tasks associated with the current anomaly model:
- **Build** — initial model training
- **Refresh** — incremental anomaly discovery on new data
- **Reprocess** — full re-evaluation of historical data

For each task you can see its status, last execution time, and result logs.

To run a reprocess immediately, click **Run Reprocess Task** in the upper-right corner of the tab.

The refresh job can run as frequently as every hour, making anomaly discovery near-real-time. See <DocLink product={props.product} path="anomaly/refresh-reprocess">Refresh &amp; Reprocess</DocLink> for configuration details.
