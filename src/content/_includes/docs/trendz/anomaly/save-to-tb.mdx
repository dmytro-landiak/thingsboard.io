When anomaly saving is enabled, Trendz writes the **anomaly score** and **anomaly score index** as time-series
telemetry to each device in ThingsBoard. This makes anomaly data available for dashboards, rule chains, and alerts
inside ThingsBoard without any additional integration work.

```plaintext
  Trendz Anomaly Model (READY)
         │  Refresh or Reprocess runs
         ▼
  ┌─────────────────────────────────┐
  │  Compute anomaly score          │
  │  Compute anomaly score index    │
  │  Aggregate at storage frequency │
  └────────────────┬────────────────┘
                   │  ThingsBoard API
                   ▼
  ┌─────────────────────────────────────┐
  │  Device telemetry                   │
  │  _EAD_<key>  → anomaly score         │
  │  _EID_<key>  → anomaly score index   │
  │  ts: per storage frequency          │
  └─────────────────────────────────────┘
```

## Configuration

### Step 1 — Set the telemetry key and storage frequency

Configure these in the **General Settings** of the anomaly model (Input tab):

| Setting | Description |
|---------|-------------|
| **Key** | Base name for the telemetry keys written to ThingsBoard. Two keys are created: `_EAD_<key>` for anomaly score and `_EID_<key>` for anomaly score index. |
| **Telemetry Storage Frequency** | How often scores are aggregated and written. Must be lower than the segment time range (fixed segments) or lower than `segment size × sliding step % / 100` (sliding window). |

<Aside type="caution">
  Set storage frequency to **1 hour or higher** to avoid unnecessary write pressure on ThingsBoard. Finalize both the
  key and the storage frequency before enabling telemetry saving — changing either after the first run produces
  inconsistent data or orphans the old keys in ThingsBoard.
</Aside>

### Step 2 — Enable anomaly refresh with saving

Telemetry saving is activated through the Refresh job, not independently. Open **Jobs** in the Input tab and enable
the **Enable Saving Anomaly Data** toggle.

When this toggle is on:
- Trendz automatically creates the required entity fields in ThingsBoard.
- Anomaly scores are written on every Refresh and Reprocess run.

When the toggle is turned off (or Refresh is disabled entirely):
- Writing stops immediately.
- The entity fields are disabled in ThingsBoard.
- **Data already written remains in ThingsBoard** and is not deleted.

<Aside type="note">
  Telemetry data persists in ThingsBoard even if the anomaly model is later deleted.
</Aside>

## Using the Saved Telemetry

Once anomaly scores are written to ThingsBoard, they behave like any other device telemetry:

- **Dashboards** — visualize `_EAD_<key>` and `_EID_<key>` on charts or gauges. See <DocLink product={props.product} path="anomaly/visualization">Anomaly Visualization</DocLink> for recommended widgets and dashboard patterns.
- **Rule chains** — use the saved keys in rule chain nodes (e.g. threshold checks, aggregations, notifications) to trigger custom logic beyond the built-in alarm creation.
- **Retention** — ThingsBoard manages the retention of saved telemetry according to its standard data retention policies. The data accumulates like any other telemetry unless a retention policy is configured in ThingsBoard.
