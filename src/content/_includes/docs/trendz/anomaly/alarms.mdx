Trendz creates ThingsBoard alarms automatically when anomalies are detected. Alarms are raised during
<DocLink product={props.product} path="anomaly/refresh-reprocess#anomaly-refresh">Refresh</DocLink> and
<DocLink product={props.product} path="anomaly/refresh-reprocess#anomaly-reprocess">Reprocess</DocLink> tasks,
and their lifecycle is tied directly to the anomaly that triggered them.

## Enabling alarm creation

Alarm creation is configured independently for Refresh and Reprocess runs.

**In Refresh** — open the **Jobs** popup from the Input tab and enable **Enable Anomaly Alarms Creation**.
Alarms will be raised for all anomalies detected on subsequent refresh runs.

**In Reprocess** — go to the **Tasks** tab, click **Run Reprocess Task**, and enable **Enable Anomaly Creation**.
Alarms will be created for anomalies found in the selected time range and items.

<Aside type="caution">
  Enable alarm creation only after validating the model in the Review tab. A poorly tuned model generates alarm
  storms that overwhelm operators and erode trust in the alerting system.
</Aside>

## Alarm lifecycle

Each detected anomaly maps to exactly one alarm in ThingsBoard. The alarm state follows the anomaly state:

```plaintext
  Anomaly detected
         │
         ▼
  ┌─────────────────────────────┐
  │  Score ≥ lowest threshold?  │
  └──────────┬──────────────────┘
             │ No ──▶ No alarm created
             │ Yes
             ▼
  ┌─────────────────────────────┐
  │  Alarm created (ACTIVE)     │
  │  Severity = matching tier   │
  └──────────┬──────────────────┘
             │
     ┌───────┴────────┐
     │ Anomaly ends   │ Anomaly deleted
     ▼                ▼   (reprocess / rebuild)
  ┌────────┐    ┌────────────────────────────┐
  │ CLEARED│    │ Cleared or Deleted         │
  └────────┘    │ (per "Alarm behavior on    │
                │  anomaly deletion" setting)│
                └────────────────────────────┘
```

**Lifecycle rules:**

| Rule | Detail |
|------|--------|
| One anomaly → one alarm | Each anomaly has exactly one linked alarm. |
| One active alarm per device | Multiple alarms can exist for a device, but only one is active at a time. |
| Alarm active while anomaly is active | As long as the anomaly persists, the alarm remains active. |
| Alarm cleared when anomaly ends | When the anomaly is no longer active, the alarm is cleared automatically. |
| Score below lowest threshold | No alarm is created — the anomaly is recorded but not escalated. |
| Anomaly deleted | The linked alarm is either cleared or deleted, depending on **Alarm Behavior on Anomaly Deletion** in the model settings. |

## Alarm configuration

Alarm type and severity thresholds are set in the **Input** tab under **Alarm Configuration**:

| Setting | Description |
|---------|-------------|
| **Alarm Type** | Category used for filtering and routing alarms in ThingsBoard. |
| **Alarm Behavior on Anomaly Deletion** | What happens to the linked alarm when the anomaly is deleted: cleared or deleted. |
| **Severities** | Score thresholds mapped to severity levels (Critical, Major, Minor, etc.). Available after the first model build — Trendz auto-generates defaults from the score distribution. |

Severity is determined at alarm creation time by comparing the anomaly score against each configured threshold. Only one severity tier is assigned — the highest matching threshold — not one alarm per tier. If the score falls below the lowest threshold, no alarm is created for that anomaly.

The **Alarm Type** is shared across all severity levels.

See <DocLink product={props.product} path="anomaly/build#alarm-configuration">Model Properties — Alarm Configuration</DocLink>
for how to configure these settings.

<Aside type="note">
ThingsBoard rule chains can subscribe to these alarms like any other ThingsBoard alarm — use them to trigger notifications, API calls, or other automation.
</Aside>
