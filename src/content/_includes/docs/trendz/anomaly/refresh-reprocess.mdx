After building an anomaly model, two operations keep it working over time:

```plaintext
  ┌──────────────────────────────────────────────────┐
  │  Anomaly Model (READY)                           │
  └─────────────────┬────────────────────────────────┘
                    │
          ┌─────────┴──────────┐
          ▼                    ▼
  ┌───────────────┐    ┌──────────────────┐
  │    Refresh    │    │    Reprocess     │
  │  (scheduled)  │    │  (on demand)    │
  │               │    │                  │
  │ New / ongoing │    │ Historical data  │
  │ telemetry     │    │ Any date range   │
  └───────────────┘    └──────────────────┘
```

- **Refresh** — a recurring scheduled job that scans new telemetry and detects ongoing anomalies automatically.
- **Reprocess** — a one-time on-demand task that applies the trained model to a specific historical time range.

## Anomaly Refresh

Refresh enables continuous, automatic anomaly discovery on live data. Configure it by clicking the **Jobs** button in
the Input tab (available only for **READY** models).

**What Refresh scans:** each run looks back at least one segment length, and extends further back to where the last
active anomaly started. This ensures Trendz can correctly determine whether an ongoing anomaly is continuing or has
ended.

| Setting | Description |
|---------|-------------|
| **Refresh Frequency** | How often the job runs, e.g. every 1 hour, every 1 day. |
| **Execute at beginning of time unit** | When enabled, the job launches at the start of the selected unit. Example: *EVERY 1 MONTH* → runs at 00:00 on the first day of each month. |
| **Choose Items** | Devices or assets on which refresh will be performed. Can include **all** items — not limited to the subset used for training. |
| **Enable Saving Anomaly Data** | When enabled, anomaly score and anomaly score index are written to ThingsBoard using the telemetry key configured in General Settings. |
| **Enable Anomaly Alarms Creation** | When enabled, alarms are raised in ThingsBoard for detected anomalies. |

<Aside type="note">
  **Refresh frequency ≠ telemetry saving frequency.** If refresh runs every 1 day and telemetry saving frequency is
  1 hour, the job runs once per day but writes 24 score points and 24 score index points per device.
</Aside>

<Aside type="caution">
  Refresh is automatically disabled when the anomaly model is rebuilt. Re-enable it manually after the rebuild
  completes.
</Aside>

## Anomaly Reprocess

Reprocess applies the trained model to historical data for a selected time range and set of items. Use it to validate
model quality on different periods or to backfill anomaly data after model changes. There is no maximum date range —
reprocess can span years of historical data.

Access it from the **Tasks** tab by clicking **Run Reprocess Task**.

| Setting | Description |
|---------|-------------|
| **Date Range** | Time range of historical data to process. No maximum limit — can span years. |
| **Items** | Devices or assets to include in the reprocess run. |
| **Enable Anomaly Creation** | When enabled, alarms are automatically created for anomalies found during reprocess. Depending on the alarm configuration, previously created alarms for the selected range may be deleted or cleared. |
| **Enable Saving Telemetry** | When enabled, anomaly scores are saved to ThingsBoard. Only available if the refresh job has *Enable Saving Anomaly Data* turned on. Existing telemetry is never deleted — it is overwritten with new scores. |

<Aside type="caution">
  Reprocess **overwrites** all previously saved anomalies for the selected time range and items under the current model.
  Existing anomaly telemetry is overwritten (never deleted). Alarm data for the range may be deleted or cleared based
  on the alarm configuration.
</Aside>
